### ISSUE #357 [CLOSED] — Expose converged per-token coordinates + assignments from sae_manifold_fit (+ warm-start init) for amortized-encoder distillation
Author: SauersML   Created: 2026-05-29T02:14:13Z   Labels: enhancement
------------------------------------------------------------------------
BODY:
## Context
Building an **amortized Manifold-SAE**: a feedforward encoder predicts, per token, the sparse atom assignments and the on-manifold coordinates `t`, so inference is a single forward pass (needed for LLM scale: F~100K atoms, billions of tokens). The gam joint Arrow-Schur solve (`sae_manifold_fit`) is the natural *teacher* — it produces the optimal coordinates/assignments — and the encoder amortizes it (Predictive-Sparse-Decomposition / iterative-amortized-inference style). For that to work, the fit needs to expose its converged latents and accept a warm start.

## Requests
1. **Expose the converged per-token latents from `sae_manifold_fit`.** After the joint solve, return (or attach to the result) the per-token, per-atom **on-manifold coordinates `t*` (N, K, d)** and the **assignments / gate `a*` (N, K)** that the solver converged to — not just the decoder blocks `B_k`. These are the supervision targets for the amortized encoder.
2. **Warm-start + bounded refinement.** Let `sae_manifold_fit` accept an initial guess for coordinates `t_init` and assignments `a_init` (encoder predictions) and run a small bounded number of refinement steps (`n_iter=k`), so training can do "encoder predicts → solver refines a few steps → distill the gap." (`Z=` exists in the signature — please document whether it is this warm-start, and for which quantity.)
3. **Standalone per-atom projection** `project(x, atom_k) -> t` (ambient point → its on-manifold coordinate for atom k). This is the minimal teacher signal for the coordinate head and is useful without running the full joint solve.

(1)+(2) together enable the amortized-inference training loop; (3) is a lighter-weight fallback teacher. Blocked in practice by #356 (the fit currently panics), but these are the interface additions needed once that lands.

## Code-grounded refinement (after reading the solver)

The warm-startable, bounded-step core for this **already exists** — it just isn't reachable from Python. Concretely, in `crates/gam-pyffi/src/lib.rs`:

- `sae_manifold_fit_inner` (~L9417) already accepts the full warm-start state as inputs: `decoder_coefficients`, `initial_logits`, `initial_coords`, and honors `max_iter` (so `max_iter=1..3` = bounded refinement). It is a plain `fn`, **not** a `#[pyfunction]`.
- The only Python-reachable fitters — `sae_manifold_fit_minimal` (~L11072), `_ibp`, `_auto` — **cold-seed every call**: PCA seed via `sae_pca_seed_initial_coords` + least-squares decoder via `sae_decoder_lsq_init`, discarding any prior state.
- The inner result dict sets: `atoms`, `logits`, `assignments_z`, `atom_active_mask`, `fitted`, `log_ard`, `log_lambda_smooth`, `reml_score`, `active_dim`, `basis_centers`, `basis_kind`, `assignment_prior`. **Missing: the converged per-token coordinates** (the `initial_coords` input has no `coords`/`t` counterpart in the output), so positions can't be threaded back even today.

### Concrete ask
Expose a stateful continue entry — e.g. `sae_manifold_fit_continue(z, state, max_iter, ...)`, or `warm_state=` + `return_state=` flags on the existing minimal fitter — that:

1. accepts prior state (`atoms`/decoder blocks, per-token `coords`, assignment `logits`, `log_ard`, `log_lambda_smooth`) and **skips the PCA + lsq cold seed when it's provided**;
2. runs a small `max_iter` (already supported);
3. returns the **full** state, including per-token coordinates, so it can be fed straight back on the next minibatch.

### Why
This turns the joint Arrow-Schur solve from a batch cold re-solve into a **per-minibatch refinement that folds into a training loop** — the M-step of an amortized-EM SAE where a feedforward encoder is distilled to the warm-continued solve, instead of running a full solve as a separate phase. The Rust core already does the optimization; this is plumbing at the FFI boundary (expose the inner driver + return coords + honor a warm seed).

Sequencing: blocked behind #356 (the inner path currently panics with an `[N*D, M] -> [N, M]` broadcast); complements #358 (the same continue entry needs GPU + minibatch to be useful at scale).

### ISSUE #393 [CLOSED] — sae_manifold_fit: index-out-of-bounds Rust panic whenever feature dim p > tangent dim d (htbeta forward operator output-length contract violation)
Author: SauersML   Created: 2026-05-29T15:09:38Z   Labels: 
------------------------------------------------------------------------
BODY:
## Trigger
`sae_manifold_fit` panics with an `index out of bounds` Rust abort for **any** atom topology whenever the decoder output dimension `p` (number of feature columns of `Z`) exceeds the per-row tangent block dimension `d = K + Σ_k d_atom_k`. Since real feature matrices have `p ≫ d`, this means the closed-form SAE crashes on essentially all non-toy inputs. It only happens to work when `p == d`.

## Minimal deterministic repro
```python
import numpy as np, gamfit
assert gamfit.__version__ == "0.1.135"
rng = np.random.default_rng(0)
n, p = 200, 5                      # p=5 > tangent d = 1(assign)+1(coord) = 2
th = rng.uniform(-np.pi, np.pi, n)
F = np.stack([np.cos(th), np.sin(th)], 1)
Z = F @ rng.standard_normal((2, p)) + 0.01 * rng.standard_normal((n, p))
m = gamfit.sae_manifold_fit(Z, K=1, atom_topology="circle", d_atom=1, n_iter=20)
```

```
thread '<unnamed>' panicked at src/terms/sae_manifold.rs:1654:17:
index out of bounds: the len is 2 but the index is 2
pyo3_runtime.PanicException: index out of bounds: the len is 2 but the index is 2
```

## Observed vs expected
- **Observed:** hard Rust `PanicException` (process-level abort, not a recoverable `GamError`) whenever `p > d`.
- **Expected:** a finite fit (or at worst a clean Python-level error). The fit must reconstruct `p`-dimensional features regardless of the tangent dimension `d`.

The crash threshold is exactly `p > d` and is topology-independent:
- `circle K=1 d_atom=1` (d = 1+1 = 2): `p=2` OK, `p=3` panics.
- `sphere K=1 d_atom=2` (d = 1+2 = 3): `p=3` OK (this is what the only sphere integration test uses), `p=4` panics.

## Root cause (current main)
The SAE installs its matrix-free cross-block `H_tβ` forward operator at `src/terms/sae_manifold.rs:2904-2916` by wiring `SaeKroneckerRows::apply_jbeta` directly as the `forward` closure of `ArrowSchurSystem::set_row_htbeta_operator`.

But the two have **incompatible output-length contracts**:

- `set_row_htbeta_operator` (`src/solver/arrow_schur.rs:1885`) documents: *"`forward` must write `out = H_tβ^(row) x` for `out.len() == d` and `x.len() == K`"* — i.e. the output is the **tangent block** of length `d = row_dims[i]`.
- `SaeKroneckerRow::apply_jbeta` (`src/terms/sae_manifold.rs:1582-1586`) is documented as *"projects the full `K·p` beta vector down to the `p`-dimensional decoded output space"* — its `u_out` loop writes indices `0..p` (the **ambient/decoded** dimension `p`), not the tangent dimension `d`.

The solver allocates the output buffer to the tangent dim (`local_i = local.slice_mut(s![..di])` in `schur_matvec`-style reduction, `src/solver/arrow_schur.rs:4046-4054`, with `di = row_dims[i] = d`), then calls `sys_htbeta_apply_row` → `op(row_idx, x, local_i)` → `apply_jbeta(row, x, out=local_i)`. `apply_jbeta` then writes `out[j]` for `j in 0..p`; when `p > di` it indexes past the length-`d` buffer and panics (the failing write is the `u_out[j] += …` line of `apply_jbeta`).

The forward operator should produce the **tangent-space** image `H_tβ x` (length `d`), which is `L · (J_β x)` — i.e. `apply_l` composed onto `apply_jbeta`'s `p`-vector output — not `apply_jbeta` alone. As wired, the `p`-length decoded output is dumped straight into a `d`-length tangent buffer, so the operator is only dimensionally valid in the degenerate `p == d` case.

Confirmed present in current main: src/terms/sae_manifold.rs:2911 forward op installs `apply_jbeta` (writes `0..p`) into the `out.len()==d` slot promised by src/solver/arrow_schur.rs:1885 and filled by src/solver/arrow_schur.rs:4046-4054 (HEAD 49366bfe7).

_Found via released-gamfit 0.1.135 fuzzing, confirmed against current main (wave 2)._

BODY:
## Summary
**Correctness drift already visible.** The sphere chart basis and its first-derivative jet are implemented in both core Rust and PyFFI, and the two have diverged in how they gate saturated-latitude derivatives.

## Locations
- `src/terms/sae_manifold.rs:441-507` — `SphereChartEvaluator::evaluate`

- Add a parity test comparing basis + jet values at saturated and non-saturated latitudes across both paths.

## Expected outcome
Single source of derivative truth for the sphere chart; eliminates the existing drift.

## Priority
**Priority 1 in cleanup order** — correctness risk already present.

### ISSUE #494 [CLOSED] — AST cross-language mirror: Crosscoder per-layer R² reimplemented in numpy; Rust owns sae_manifold_reconstruction_r2

## AST cross-language mirror: Crosscoder per-layer R² reimplemented in numpy; Rust already owns `sae_manifold_reconstruction_r2`

**Method:** op-signature extraction flagged `gamfit/crosscoder.py:176 fit` as a heavy-math Python function (arith=17, loops, `{mean,sqrt,sum,abs}`). Reading it, the final block recomputes a reconstruction R² in numpy that the Rust core already exposes verbatim through the FFI.

### The duplicated math (verified by reading)

`gamfit/crosscoder.py:287-292`, per layer:
```python
centered = tl_np - tl_np.mean(axis=0, keepdims=True)
ss_tot = float(np.sum(centered * centered))
ss_res = float(np.sum((tl_np - rl_np) ** 2))
per_layer[li] = (1.0 - ss_res / ss_tot if ss_tot > 0.0 else float("nan"))
```
This is column-mean-centered SST, residual SSR, `1 - SSR/SST`, NaN when SST==0.

### The Rust core that should own it

`crates/gam-pyffi/src/lib.rs:11891 sae_manifold_reconstruction_r2(observed, fitted)` (registered at lib.rs:22777) computes the identical quantity:
```rust
col_means[col] = Σ_row observed / n_rows;          // per-column mean
ssr += d*d  (d = observed - fitted);                // residual SS
sst += dm*dm (dm = observed - col_means[col]);      // total SS, mean-centered
if sst == 0.0 { return NAN }
Ok(1.0 - ssr / sst)
```
Same column-mean centering, same SSR/SST, same `1 - ssr/sst`, same SST==0 → NaN contract. The crosscoder loops it per layer, but each iteration is exactly this kernel.

This violates "Python is a thin wrapper; all math in Rust." The R² statistic is a numeric kernel Rust already owns and exports; the crosscoder should call it per layer rather than re-deriving the formula in numpy (and risking drift from the canonical SST==0/non-finite handling, which the Rust version guards explicitly and the Python copy does not).

### Fix (route through FFI)

In `gamfit/crosscoder.py` fit (and `per_layer_r2`), replace the inline `centered/ss_tot/ss_res` block with a per-layer call to `rust_module().sae_manifold_reconstruction_r2(tl_2d, rl_2d)`, where `tl_2d`/`rl_2d` are the `(N, d_l)` observed/fitted layer matrices already in hand at lib.rs use site. The Rust fn takes `(observed, fitted)` 2-D arrays and returns the scalar R² with the SST==0 → NaN and non-finite guards.

### ISSUE #596 [CLOSED] — [SAE] Compact active-set Newton step ignores step_size for manifold coordinates
Author: SauersML   Created: 2026-06-02T14:30:14Z   Labels: bug
------------------------------------------------------------------------
BODY:
**Verified — CONFIRMED.**

In `apply_newton_step_impl` (`src/terms/sae_manifold.rs`), the **compact** active-set branch writes coordinate deltas without the line-search `step_size` factor, while logits and the **dense** branch both apply it.

Compact branch (~`src/terms/sae_manifold.rs:4589-4594`):
```rust
delta_coord[row * d + axis] = full_delta[row_base + axis];   // NO step_size
```
Logits in the same branch (~4581) do scale: `step_size * full_delta[...]`.

Dense branch (~4621-4626):
```rust
delta_coord[row * d + axis] = step_size * delta_ext_coord[row_base + axis];   // WITH step_size
```

**Why it's wrong:** a damped Newton/Armijo step must apply θ_new = retract(θ_old, α·Δθ) with α = step_size uniformly. The compact path damps logits/β but takes a full-size coordinate move, so line-search accept/reject decisions are made on a step that is not the actual scaled Newton step. Compact layout is the path used for JumpReLU and large-K sparse plans, so this is on the important paths.

**Fix:** `delta_coord[row*d+axis] = step_size * full_delta[row_base+axis];` and add a regression test: with step_size=0.25, compact coordinate displacement norm must be 0.25× the full Newton displacement (matching dense).

_Source: static SAE/manifold-SAE audit; verified against current code at commit 3c1a4efa2._

- **Softmax, K=1:** 1.0 is correct (the only simplex point).
- **IBP, K=1, logit=0, temp=1, prior=1:** correct gate is sigmoid(0)=**0.5**, code returns 1.0.
- **JumpReLU, K=1, logit≤threshold:** correct gate is **0.0**, code returns 1.0.

Worse, the prior-gradient path only skips the softmax K==1 case, so with IBP/JumpReLU the optimizer can move a logit the reconstruction ignores — a free parameter affecting only the prior.

**Fix:** gate the shortcut on softmax only —
```rust
if self.k_atoms() == 1 && matches!(self.mode, AssignmentMode::Softmax { .. }) { ... }
```
and dispatch IBP/JumpReLU through their real assignment functions for K=1. Add tests: IBP K=1 logit=0 ⇒ 0.5; JumpReLU K=1 below threshold ⇒ 0.0.

```rust
if logits[i] > threshold {

```

- (B) Implement true JumpReLU `if z>θ {z} else {0}` with a centered surrogate derivative for optimization (see #threshold-centering issue) and L0-surrogate penalty.

Gated-SAE (separate gate × softplus magnitude) is a distinct family — don't conflate.

- Parity test: threshold=2.0 ⇒ logit 1.9 inactive / 2.0 boundary / 2.1 active, torch == closed-form.

### ISSUE #600 [CLOSED] — [SAE] JumpReLU inactive atoms are absorbing/dead — cannot reactivate during the solve
Author: SauersML   Created: 2026-06-02T14:30:51Z   Labels: bug
------------------------------------------------------------------------
BODY:
**Verified — CONFIRMED (all three sub-claims).**

1. Compact layout keeps only atoms with `row_logits[k] > threshold` (~`src/terms/sae_manifold.rs:2315-2318`).
2. Arrow-Schur **always** uses compact layout for JumpReLU (~3047-3054).
3. Logit JVP (`if logit_k <= threshold { return; }` ~5860) and prior-grad (`if logit <= threshold { continue; }` ~6060) both skip inactive atoms.

Consequence: once `logit ≤ threshold`, the atom's assignment is 0, its reconstruction Jacobian is 0, its prior gradient is 0, and it is excluded from the compact Newton system → its computed update is exactly zero. It can never re-cross the threshold. The PyFFI cold start seeds all logits positive (~11508-11543), which only hides the cold-start failure; any later deactivation is permanent. This is absorbing active-set pruning, not a learnable JumpReLU/STE SAE.

**Fix options:**
- (A) Keep all logits in the local state (dense logits); apply hard support only in the forward value, keep surrogate gradient nonzero near threshold.
- (B) Candidate active set: include atoms with `|logit-threshold| < c·temperature`, top-m by surrogate gradient, or recently active.
- (C) Outer active-set loop allowing reactivation between iterations after a smooth warm-up.

Test: start with logit < threshold on data that needs the atom; assert it can become active.

_Source: static SAE/manifold-SAE audit; verified against current code at commit 3c1a4efa2._

Fixed on main in `484cb83f3`: a reactivation band `jumprelu_in_optimization_band(logit, threshold, temperature) = logit > threshold - JUMPRELU_REACTIVATION_MARGIN*temperature` (margin 4τ) replaces the hard absorbing filter, so below-threshold atoms can re-cross.

### ISSUE #602 [CLOSED] — [SAE] Product/cylinder intrinsic coordinates beyond the first are ignored in the torch manifold basis
Author: SauersML   Created: 2026-06-02T14:31:15Z   Labels: bug
------------------------------------------------------------------------
BODY:
**Verified — CONFIRMED.**

Config permits product `intrinsic_rank >= 2` and cylinder `== 2` (`gamfit/torch/manifold_sae.py:263-270`), but `_basis_rust()` evaluates only the first coordinate:
```python
return self._periodic_features(t[:, :1])   # ~ gamfit/torch/manifold_sae.py:380-385
```
(comment: remaining intrinsic coords \"enter through the amplitude path\"). `forward` uses positions only via `_eval_basis_on_manifold(positions, ...)` (~701-705); the amplitude path comes from `amp_logits`, not `positions[..., 1:]` (~706-708).
