## P1. `src/terms/basis.rs` lint break — blocks Test, Clippy, Rustfmt, and every workflow that calls `cargo build` first

* **Where**: `src/terms/basis.rs:7720-7770` (the freshly added
  anisotropic closed-form psi-derivative substitution block).
* **Affected jobs**: `Test` (run 25202448482, job 73896114577),
  `Clippy` (job 73896114565), `Rustfmt` (job 73896114568, 80+ diffs in
  `src/inference/predict.rs:4971..5692`).
* **Evidence (Test job)**:
  ```
  error: variable `op1_cf_cross_raw` is assigned to, but never used
      --> src/terms/basis.rs:7720:9
  error: variable `op2_cf_cross_raw` is assigned to, but never used
      --> src/terms/basis.rs:7721:9
  error: value assigned to `op1_cf_cross_raw` is never read
      --> src/terms/basis.rs:7769:9
  error: value assigned to `op2_cf_cross_raw` is never read
      --> src/terms/basis.rs:7770:9
  error: variable does not need to be mutable
      --> src/terms/basis.rs:7729:13   (closure binding `let mut substitute = …`)
  could not compile `gam` (lib) due to 5 previous errors
  ```
* **Root-cause hypothesis**: the anisotropic closed-form psi pullback
  was wired up to compute per-axis cross terms but the cross matrices
  produced by `substitute(1, &mut op1_info)` and `substitute(2, &mut
  op2_info)` are stored in two locals (`op1_cf_cross_raw`,
  `op2_cf_cross_raw`) that are then **never consumed** — the rest of
  the function still reads `op1_info.s_first` / `s_second` / `s_first_raw`
  populated as side-effects. Either the cross-derivative path is
  half-wired (the consumer at the assembly site for op1/op2 second
  derivatives is missing) or the locals are dead by design and should
  be removed. With `#![deny(unused_variables)]` in `src/lib.rs:2` this
  is a hard error rather than a warning.
* **Why P1**: every `cargo build` step in CI fails here, so Clippy,
  Rustfmt, and the Test workflow can never reach the test runner.
  This is also gating the next biobank rerun's compile step. Fix
  this first before debugging the runtime regressions below.
* **Suggested fix scope**: confirm whether the anisotropic op1/op2
  closed-form cross-derivative outputs are needed by a downstream
  caller. If they are, route the returned `Vec<Vec<Array2<f64>>>` into
  the second-derivative assembly site (likely the `op1_info` /
  `op2_info` second-derivative branch a few hundred lines below the
  closure). If not, the closure should not return them and the locals
  should not exist. Either way, do **not** silence with `_op1…` /
  `#[allow(unused_variables)]` — per project memory
  (`feedback_no_dead_code_shortcuts.md`) the unused item must be
  resolved structurally. Also rerun `cargo fmt` over `src/inference/predict.rs`
  in the same commit.

## P2. Biobank-scale wallclock timeouts on `*_duchon60` / `*_tps60` (rust_gam)

* **Where**: outer REML loop on dense Duchon / TPS / Matérn at p≈60,
  n≈300K, before the first BFGS step completes within
  `BENCH_CMD_TIMEOUT_SEC=2400`.
* **Affected matrix configs** (run 25199437327):
  * `rust_gamlss_additive_duchon60` (job 73888112719) — last heartbeat
    `elapsed=2388.1s`, then `command-timeout rc=124 timeout_sec=2400`
    while still inside the same `gam fit --predict-noise smooth(age_entry_std)
    + smooth(lat_final_std) + smooth(lon_final_std) + pc1..pc4` invocation
    at RSS=6.31 GiB / cpu≈125%.
  * `rust_gam_additive_duchon60` (job 73888112724) — same pattern.
  * `rust_gam_additive_tps60` (job 73888112727) — same pattern.
  * `rust_gam_additive_matern60` (job 73888112729) — slightly different:
    one full BFGS finished, then the second-stage outer REML is iterating
    `[STAGE] outer eval` ValueAndGradient calls of ~26-35s each on
    dim=9 with `|g|≈4.24e1` plateau and repeated
    `[WARN] [STAGE] SLQ tiny-system Cholesky failed (dim=101);
    recovered logdet via symmetric eigendecomposition with
    positive-eigenvalue floor 1.255e-8` and
    `[WARN] [BFGS Adaptive] Strong Wolfe failed at iter 3. Falling back
    to Backtracking.` — i.e. the outer is bouncing on a noisy/almost-flat
    surface and never reduces `|g|` below ~42, eventually hitting the
    2400s cap.
* **Root-cause hypothesis**: the duchon60/tps60 path is doing dense
  per-eval work that scales like O(n·p²) or worse without the
  matrix-free routing. The Matérn job in particular shows two
  symptoms simultaneously:
  1. SLQ tiny-system Cholesky fallback (`src/linalg/utils.rs:497`) on
     a 101×101 system with `λ_min≈10⁻⁸` — i.e. the SLQ probe matrix
     is rank-deficient, so logdet is being taken via symmetric eig
     each iter. That recovery is correct but expensive when it
     fires every outer eval.
  2. Strong Wolfe failures and a flat `|g|≈4.24e1` floor: the
     gradient floor is consistent with the
     `survival_marginal_slope_hang_root_cause.md` pattern in memory
     — gradient is mis-scaled relative to the parameter natural scale,
     so BFGS oscillates and the
     `outer_scaled_tolerance(τ, V_seed)` in
     `scale_invariant_convergence.md` cannot certify stationarity.
  3. The `gamlss_additive_duchon60` job stays at 6.31 GiB RSS for
     the full 2400 s without any `[STAGE]` activity printing — the
     process is plausibly stuck in a single non-yielding inner
     P-IRLS solve or in dense `weighted_gram` assembly. The recent
     fix `ctn_matrix_free_workspace.md` removed dense `weighted_gram`
     for CTN, but this job uses **GAMLSS-binomial** (separate=true),
     not CTN. `gamlss_matrix_free_workspace.md` says all four GAMLSS
     location-scale families have matrix-free workspaces, but the
     plain GAMLSS-binomial-additive code path may still go through a
     dense joint Hessian build at p=60.
* **Pointer files**:
  * `src/solver/reml/eval.rs`, `src/solver/reml/runtime.rs` —
    outer ValueAndGradient bridge; check whether
    `OuterProblem::with_dense_hessian_work_hint(n·p²)` (per
    `work_cost_routing.md`) is being supplied for these duchon60/tps60
    configurations (the n·p² gate is what auto-routes to BfgsApprox
    instead of Analytic+Analytic).
  * `src/families/binomial.rs` and the GAMLSS-binomial joint-H
    assembly — verify `ExactNewtonJointHessianWorkspace` is on the
    additive (single-separate) path, not just on location-scale.
  * `src/terms/smooth.rs` and `src/terms/basis.rs` — Duchon/TPS
    centers/penalty cache for n=300K.
* **Why P2**: four of six biobank-scale failures are this category, and
  the SLQ + Wolfe noise points at a specific algorithmic regression
  rather than wallclock crowding.

## P3. Anisotropic margslope-Duchon-16d non-monotonicity blow-up (`-9.328e188`)

* **Where**: `gam fit` exits during the prediction-time monotonicity
  certificate at `src/inference/predict_input.rs:335`.
* **Affected matrix configs**:
  * `rust_margslope_aniso_duchon16d_linkwiggle_scorewarp_fast`
    (job 73888112735, run 25199437327).
  * `rust_survival_margslope_aniso_duchon16d_gm_linkwiggle_scorewarp_timewiggle_fast`
    (job 73888112744, run 25199437327).
* **Evidence**:
  ```
  03:03:29 [INFO] [BFGS] step accepted via StrongWolfe; inverse update applied
  03:03:29 [WARN] [BFGS] Max iterations reached: iters=3, f=2.583096e3,
                       ||g||=2.069e0, fe=21, ge=11, Δ=4.541e-2
  03:03:30 [INFO] [STAGE] CTN monotonicity certificate hit |A|=1
                  m_inactive=1.137e-2 α_A·L=-1.211e-15 → α=-2.790e-16
                  (skipped n·…)
  03:03:34 error: prediction failed: transformation-normal fit is non-monotone
                  in y for at least one observation. Analytic lower bound on
                  h'(y, x) is -9.328e188, threshold ...
  ```
* **Root-cause hypothesis**: the BFGS outer is being terminated on
  `Max iterations reached` after only 3 iters (`||g||=2.069e0`, not
  near KKT) because the seed-cap cascade (`biobank_outer_inner_cost_fix.md`)
  caps iterations to keep wallclock tractable. The **fit** result is
  therefore far from the optimum on this 22-dim hyperparameter
  surface; the fitted CTN has h′ that tunnels into a regime where the
  closed-form lower bound `inf h'(y,x) = -9.328e188` (clearly
  numerically meaningless — exponent overflow). Two distinct bugs are
  conflated:
  1. **Outer convergence**: BFGS terminating with `||g||=2.069` on a
     22-dim aniso surface and the certificate-skip diagnostic
     `α_A·L=-1.21e-15` (active set is of measure zero on the grid)
     suggests the active-set / KKT bookkeeping in
     `src/families/transformation_normal.rs` ~line 3756 (the
     `CTN monotonicity certificate hit` block) is reporting a
     pseudo-convergence. The runtime believes it is feasible
     (`α≈-2.79e-16`, basically zero) so it accepts the step, but
     the fit object then fails the tighter prediction-time check.
  2. **Numerical blow-up**: the lower bound `-9.328e188` cannot be a
     true derivative — this is a sign of catastrophic cancellation
     or λ_aniso pulled to extreme values (the per-axis `aniso_log_scales`
     touched the operator-log-λ box anchor described in
     `spatial_adaptive_baseline_anchored_box.md`). The closed-form
     evaluator at `closed_form_aniso_psi_derivatives_in_total_basis`
     (in `src/terms/basis.rs` near line 7737, where the broken
     locals from P1 live) is producing extremely large second
     derivatives.
* **Pointer files**:
  * `src/families/transformation_normal.rs` (CTN cert + monotonicity
    around lines 3700-3780).
  * `src/inference/predict_input.rs:335` (monotonicity check that
    raises the user-visible error).
  * `src/terms/basis.rs:7700-7900` (anisotropic closed-form path —
    same block whose lints break P1; once the cross matrices are
    correctly consumed, the second-derivative magnitudes likely fix
    themselves).
* **Why P3**: two of six biobank-scale failures are this category and
  the symptom is plausibly downstream of P1 (the same anisotropic
  closed-form block).

  `seed=70 n=20 stacked/gaussia`, `seed=133 n=25 regime/heteros`,
  `seed=119 n=30 surface/lognorm`, `seed=123 n=30 low_dim/contami`,
  `seed=127 n=30 regime/bimodal`, `seed=51 n=20 stacked/laplace`,
  `seed=110 n=500 surface/t`, `seed=131 n=100 regime/mixture`,
  `seed=107 n=50 low_dim/ar1`, `seed=105 n=100 stacked/t`,
  `seed=121 n=80 additive/lognorm`, `seed=81 n=500 low_dim/periodic`,
  `seed=65 n=64 regime/gaussia`.
* **Root-cause hypothesis**: this is `seq()` or `range()` in the R
  mgcv harness consuming a non-finite value — likely a smoothing
  parameter or a knot computed from a degenerate `x` (e.g. all-zero
  after standardization, or NA from `signal=stacked`/`regime`). It is
  **not** a rust bug, but the gate evaluator counts these as
  unevaluated (`[M:ERR]`) and the `coverage` gate fires when too many
  trials are unevaluated. The `coverage` gate appears in the gate
  fail line.
* **Pointer files**: `bench/` harness (R side), specifically the
  mgcv driver script and the seed/data generator. (Not Rust-side.)
* **Why P5**: lowest priority — fixing P4 (the rust quality gates)
  will not change this category, but it pads the failure count and
  may need its own filter so that the rust gates aren't muddled with
  R-side aborts.
